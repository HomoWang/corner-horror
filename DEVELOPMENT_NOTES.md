# 開發紀錄

## 2026-07-14：雙螢幕操作重整

### 實測發現

- 把來電、劇情、密碼鍵盤與選項放在手機，會迫使玩家反覆低頭，直接破壞大螢幕驚嚇的
  注意力設計；手機必須是盲操作控制器。
- WebSocket 的 `pointerdown` 與 `pointerup` 可能在大螢幕同一動畫幀前一起到達。若只讀取
  最後的布林狀態，快速點擊會被吃掉；主機端因此保留一次至少一幀的 action pulse。
- 原本的投影換算只把光錐移到畫面中央約一半範圍，加上 vignette 後左右邊界幾乎全黑。
  水平投影靈敏度已由 `0.5` 提高至 `0.72`，邊界限制改為 `1.5%–98.5%`，同時保留最低
  邊緣可見度。
- 大螢幕的 Web Audio 不能由手機送來的 WebSocket 訊息解鎖，因此不能把主機音效當成
  開始條件。手機中央鍵會直接開始影像，並由已透過手機觸碰解鎖的手機音效引擎播放完整
  背景音與驚嚇音；主機音效只保留為可選同步輸出。

### 現行設計決定

- 手機只顯示一顆大型中央互動鍵；重新對準為底部次要功能。
- 來電、文字、`0317`、左右抉擇與重玩全部顯示在大螢幕。
- 密碼以四次中央鍵依序輸入；結局選擇以光錐指向左右，再按中央鍵確認。
- 左窗第一個主要驚嚇改為全屏恐怖臉、主機尖叫／衝擊、雷聲與手機震動同步觸發。

## 2026-07-14：恐怖節奏與照片檢視

- 來電畫面與獨立 cue 都會要求手機播放鈴聲，音效引擎會去除一秒內的重複播放；未接聽時
  每 5.6 秒再響一次，避免一次網路訊息遺失後整段無聲。
- 開場等待延長至 7.2 秒；通話、窗戶餘韻、照片正反面與兩段錄音各自加入最低閱讀時間，
  避免連點把情緒曲線壓成一條直線。
- 照片改為拿起正面、翻至背面、放回牆上後異變的三段演出，並在正面加入短暫第四人影、
  背面加入身後刮擦聲與延遲衝擊。
- 房間底圖 gamma、最低環境光、光錐寬度與垂直靈敏度均提高；矮櫃另加極弱的反光提示，
  玩家仍需搜尋，但不會因黑位壓死而看不見抽屜。

## 2026-07-14：錄音帶、門前傾聽與結局差異

### 稽核發現

- 前半段已有電話、窗戶與照片的事件節奏，但抽屜之後只剩兩張文字畫面；玩家不是在看
  錄音帶，而是在讀說明，情緒會在高潮前掉下來。
- 找到房門後直接出現二選一，沒有讓「門外哭聲」和「房內腳步」形成空間矛盾；原本開門
  1.15 秒便跳臉，也太容易預測。
- 開門跳臉時若保留上一個選擇框、標題與校正按鈕，畫面會像 UI 疊圖而非電影式驚嚇。

### 實作調整

- 新增 CSS 繪製的舊錄音機、旋轉卡帶、逐句字幕、毀損標記與每段 7.2 秒進度；兩段錄音
  之間會確實重啟動畫時間。
- 新增可重複觸發的左右餘光人影、房間閃爍與短暫全黑；照片背面、錄音與門前使用不同
  組合，並保留「有聲無物」的假驚嚇。
- 找到房門後加入至少 6.2 秒的貼門傾聽狀態：敲門、背後腳步、餘光人影、手機與門外同鈴
  依序發生，之後才允許選擇。
- 開門結局延後至 2.2 秒才跳臉，先經過 1.75 秒熄燈；最後演出期間隱藏劇情、選擇與主機
  控制介面。封印結局延長為 4.35 秒，僅用燈閃、人影和呼吸收尾，不共用跳臉。

### 瀏覽器雙頁實測

- 本機 Host 與 Controller 使用同一房號實際配對，所有步驟均由 Controller 的同一顆中央鍵
  推進，Host 以滑鼠模擬手機方向；錄音帶、貼門傾聽、開門與封印兩條分支皆完整走通。
- 開門驚嚇已確認只剩全屏恐怖臉；封印高潮已確認介面全數清空後才顯示房間餘韻。
- Host 與 Controller 在兩條完整流程中皆無 console error 或 warning。
